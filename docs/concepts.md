# Понятия

Простые объяснения терминов из архитектуры.

## Diff

Diff — это список изменений в коде.

Он показывает:

- какие строки добавили;
- какие строки удалили;
- какие строки изменили;
- в каких файлах это произошло.

Пример:

```diff
- Button: Save
+ Button: Save task
```

Команда для просмотра изменений, которые ещё не подготовлены к коммиту:

```bash
git diff
```

Команда для просмотра изменений, которые уже подготовлены к коммиту:

```bash
git diff --staged
```

## Work modes

Work modes — это режимы работы агента. Они помогают агенту понять, что именно он должен делать.

- `implementation` — менять код.
- `review` — проверять изменения и сообщать о проблемах, но не редактировать файлы.
- `task-finish` — проверять, можно ли закрыть задачу, и чистить контекст после подтверждения.
- `architecture-update` — обновлять правила AI-разработки, а не код приложения.

## Project invariants

Project invariants — это правила проекта, которые нельзя ломать.

Это «несущие стены» проекта.

Пример:

```text
Microtask must belong to a Week Task.
```

Инварианты проекта должны храниться в `ai/project-context.md`, а не в `AGENTS.md` или `CLAUDE.md`.

## Skills

Skills — это переиспользуемые процедуры для конкретных задач.

Примеры:

- `ui-review` проверяет изменения интерфейса.
- `security-review` проверяет риски безопасности.
- `write-tests` решает, нужны ли автотесты.
- `task-intake` принимает новую задачу и записывает её в текущую память проекта.
- `task-finish` безопасно закрывает задачу.

Skills нужно подключать только по необходимости.

## task-finish

`task-finish` состоит из двух фаз.

Phase 1 проверяет, можно ли закрыть задачу. Файлы не редактируются.

Phase 2 чистит контекст только после подтверждения пользователя. Она обновляет:

- `ai/changelog.md`
- `ai/decisions.md`, если нужно
- `ai/current-task.md`
- подтверждённые записи в `ai/future-tasks.md`, если во время задачи появились идеи на потом

`task-finish` не должен менять код приложения. После cleanup он должен сохранить результат: push на GitHub, если GitHub подключён, или local-only fallback, если GitHub недоступен.

## architecture-update

`architecture-update` меняет саму систему разработки.

Это не то же самое, что `implementation`.

- `implementation` меняет приложение.
- `architecture-update` меняет правила, по которым AI-агенты работают над приложением.

Перед изменением файлов архитектуры агент должен показать:

```text
current rule → proposed rule → exact files
```

Затем агент обязан спросить:

```text
Replace this?
```

## Environment check

Environment check — это быстрая проверка первого запуска или новой сессии.

Агент проверяет:

- есть ли все базовые файлы архитектуры;
- есть ли все базовые skills;
- доступны ли ожидаемые внешние skills и инструменты: code-review-graph, agent-skills-for-context-engineering и Superpowers;
- есть ли controlled methodologies, например Superpowers.

Это не проверка зависимостей приложения. Например, она не проверяет, установлены ли Python-пакеты или npm-зависимости.

Если не хватает базовых skills, агент пишет, что восстановить из шаблона. Если не хватает ожидаемых внешних tools или skills, агент пишет предупреждение, но не блокирует работу. Ничего не устанавливает без подтверждения пользователя.

После проверки агент должен показать меню доступных следующих commands и skills. Это справка, а не команда запускать всё подряд.

Пример меню:

```text
Доступные следующие команды и skills
- task-switch — переключиться на другую задачу или продвинуть future task.
- task-finish — проверить, можно ли закрыть текущую задачу.
- task-intake — принять новую задачу или определить, нужен ли task-switch.
- Superpowers — разбирать баги, сложные задачи и задачи с непонятным радиусом изменений.
```

## Приоритет скиллов

Внутренняя архитектура важнее внешних skills.

Порядок такой:

1. `AGENTS.md` или `CLAUDE.md`
2. `ai/current-task.md`
3. нужный базовый skill
4. optional project skills и ожидаемые внешние skills/tools
5. controlled external methodologies

Внешний skill не должен отменять режим работы, правила подтверждения, task-finish, architecture-update и принцип чистой архитектуры.

Если нужно выбрать один инструмент для разработки, приоритет у `code-review-graph`.

Почему:
- он точнее подходит для code review;
- он считает радиус изменений;
- он помогает агенту читать только нужные файлы.

## Superpowers как контролируемая методология

Superpowers полезен для багов и больших задач, где нужно сначала уточнить идею, согласовать дизайн, сделать план, использовать TDD или разбить работу между subagents.

Но Superpowers не должен становиться процессом по умолчанию.

Он ожидается для:
- багов, регрессий, падений и performance-проблем;
- больших, расплывчатых или рискованных задач;
- задач, где нужно проектирование, TDD или subagents.

Он не используется для:
- правки текста;
- простых UI-правок;
- task-finish;
- architecture-update без явного запроса.

Главное правило:

```text
Наша архитектура управляет процессом.
Superpowers усиливает процесс только после разрешения.
```


## environment-check не является режимом работы

`environment-check` — это служебная проверка первого запуска.

Она проверяет:
- базовые файлы архитектуры;
- базовые skills;
- ожидаемые внешние tools;
- controlled methodologies.

После этой проверки агент продолжает работу в одном из режимов:
- `implementation`
- `review`
- `task-finish`
- `architecture-update`

Перед продолжением агент показывает доступные следующие commands и skills. Меню не запускает эти workflows автоматически.

Так режимы работы не смешиваются со стартовой проверкой.


## Task switching

Перед рабочей задачей агент сначала использует `task-intake`.

`task-intake` смотрит на `ai/current-task.md` и решает:

- записать первую задачу;
- продолжить текущую;
- запустить `task-switch`, если пользователь просит другую задачу;
- сохранить идею в `ai/future-tasks.md`.

`task-switch` нужен, когда текущая задача ещё не закрыта, а пользователь просит начать другую.

Он не закрывает старую задачу. Для закрытия есть `task-finish`.

Что делает `task-switch`:

1. Показывает текущую незавершённую задачу.
2. Показывает новую задачу.
3. Предупреждает о риске потери контекста.
4. Предлагает варианты:
   - продолжить текущую задачу;
   - поставить текущую на паузу и начать новую;
   - закрыть текущую через `task-finish`;
   - заменить текущую задачу.
5. Меняет файлы только после подтверждения пользователя.

Если задача ставится на паузу, краткая запись попадает в `ai/paused-tasks.md`.

Если пользователь хочет сохранить идею на потом, она попадает в `ai/future-tasks.md`, а не в `ai/paused-tasks.md`.


### Как агент понимает, что задача другая

Агент сравнивает новый запрос с `ai/current-task.md`.

Это другая задача, если:
- поменялась цель;
- поменялся work mode;
- поменялись основные файлы или область проекта;
- поменялись Done criteria;
- новый запрос не помогает завершить текущую задачу;
- новый запрос создаёт отдельный результат.

Это продолжение текущей задачи, если новый запрос уточняет, сужает, тестирует, ревьюит или завершает текущую цель.
